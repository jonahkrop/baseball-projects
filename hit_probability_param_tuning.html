Parameter Tuning¶

Last time, I wrapped up my hit probability model testing by choosing a gradient boosting algorithm. Today, I'm going to do some grid search hyperparameter tuning to improve the model as much as I can. Since I'm relatively new to this, I'm going to just do one hyperparameter at a time.

I'll start by setting up a baseline model with all default settings. Like last time, there's a few things I have to do to the data first:

1) Remove any bunts from the dataset
2) Map any non-hit outcome to 'out' (e.g. sacrifice fly, fielder's choice)
3) One-hot encode batter handedness
4) Over-sample imbalanced data

Just like before, I'm going to use imblearn to set up a pipeline for cross-validating the model. imblearn is set up to work with imbalanced datasets, so it will apply over-sampling just on the training data, and validate or test on the standard data.

I'm going to score using log loss, since I'm planning to use the probabilistic classifications from the model.

Hyperparameter Tuning¶

Like I mentioned, I'm going to do one hyperparameter at a time using grid search, evaluating using log loss. There's many parameters that can be tuned, but I'm going to focus on a few important ones:

  • learning_rate
  • n_estimators
  • max_depth
  • min_samples_split
  • min_samples_leaf

Learning Rate & Number of Trees¶

I know I said I'd do one at a time, but learning rate and tree number go hand in hand. In gradient boosting, new trees are created and added to the model to explain errors from the previous step of the model. Learning rate governs the weight each tree is given in the model. A lower learning rate gives you a lower weight for each tree -- so more trees would be necessary to maintain a given level of performance for a lower learning rate. sklearn defaults are 0.1 for learning_rate and 100 for n_estimators.

In [52]:
# set pipeline
pipe = Pipeline([
    ('smote', SMOTE()),
    ('classify', GradientBoostingClassifier(random_state=0))
])

# parameters for grid search
grid = {'classify__learning_rate': [0.2, 0.15, 0.1, 0.05, 0.01],
        'classify__n_estimators': [100, 150, 200, 250, 300, 400]}

res1 = GridSearchCV(pipe, grid, scoring='neg_log_loss',
                    return_train_score=True, n_jobs=4).fit(X, y)
results1 = pd.DataFrame(res1.cv_results_)

print('best learning rate = %s' % res1.best_params_['classify__learning_rate'])
print('best number of trees = %s' % res1.best_params_['classify__n_estimators'])
print('log loss with optimal parameters: %0.3f' % res1.best_score_)

best learning rate = 0.1
best number of trees = 250
log loss with optimal parameters: -0.759

From the grid search, the model is optimized with learning rate = 0.1 and number of trees = 250. Overall, the optimization made a relatively small improvement to log loss, which isn't too surprising since the learning rate is unchanged from the baseline model. On a side note, it's pretty clear that the number of trees is governing the model runtime, so I'm glad the grid search isn't suggesting more than 250.

Max Tree Depth¶

Max tree depth controls the number of nodes allowed for a tree. sklearn defaults this to 3, so I'll start around there and go as high as around 30.

In [24]:
grid = {'classify__max_depth': [2, 4, 6, 10, 20, 30]}

# this time, use ideal learning rate and num. trees
pipe = Pipeline([
    ('smote', SMOTE()),
    ('classify', GradientBoostingClassifier(

        learning_rate=res1.best_params_['classify__learning_rate'],
        n_estimators=res1.best_params_['classify__n_estimators'])
    )
])

res2 = GridSearchCV(pipe, grid, scoring='neg_log_loss',
                    return_train_score=True,
                    n_jobs=4).fit(X, y)

best max_depth = 4
log loss with optimal parameters: -0.750

Similar improvement here as with tuning learning rate and number of trees. Larger max depth makes a real difference for overfitting -- training error is almost 0 at a max depth past 10, and testing error gets bad.

Min samples split¶

min_samples_split governs the number of samples of data needed to split a node into leafs. The sklearn default is 2. We have a pretty large dataset so this can get relatively large.

Min samples leaf¶

Finally, we have min_samples_leaf. min_samples_leaf is similar to min_samples_split, requiring a minimum number of samples to create a leaf. The sklearn default is 1.

Adjusting min_samples_leaf has a comparable impact to min_samples_split, which isn't too surprising. Leafs are just nodes that have no additional nodes coming off them -- they make a classification instead of passing to another node.

Wrapping Up¶

Successfully completed hyperparameter tuning! I didn't get a massive improvement in log loss (-0.769 to -0.747), but I'm still happy with the results. Here's the ultimate optimal parameters:

  • learning rate = 0.1
  • number of trees = 250
  • max tree depth = 4
  • min samples to split node = 15
  • min samples to split leaf = 6
