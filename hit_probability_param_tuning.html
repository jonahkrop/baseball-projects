I'll start by setting up a baseline model with all default settings. Like last time, there's a few things I have to do to the data first:

1) Remove any bunts from the dataset
2) Map any non-hit outcome to 'out' (e.g. sacrifice fly, fielder's choice)
3) One-hot encode batter handedness

Hyperparameter Tuning¶

Like I mentioned, I'm going to do one hyperparameter at a time using grid search, evaluating using log loss. There's many parameters that can be tuned, but I'm going to focus on a few important ones:

  • learning_rate
  • n_estimators
  • max_depth
  • min_samples_split
  • min_samples_leaf

Learning Rate & Number of Trees¶

I know I said I'd do one at a time, but learning rate and tree number go hand in hand. In gradient boosting, new trees are created and added to the model to explain errors from the previous step of the model. Learning rate governs the weight each tree is given in the model. A lower learning rate gives you a lower weight for each tree -- so more trees would be necessary to maintain a given level of performance for a lower learning rate. sklearn defaults are 0.1 for learning_rate and 100 for n_estimators.

In [77]:
# parameters for grid search
grid = {'learning_rate': [0.2, 0.15, 0.1, 0.05, 0.01],
        'n_estimators': [50, 100, 200, 300, 400, 500]}

model = GradientBoostingClassifier(random_state=0)

res1 = GridSearchCV(model, grid, scoring='neg_log_loss',
                    return_train_score=True, n_jobs=4).fit(X, y)
results1 = pd.DataFrame(res1.cv_results_)

print('best learning rate = %s' % res1.best_params_['learning_rate'])
print('best number of trees = %s' % res1.best_params_['n_estimators'])
print('log loss with optimal parameters: %0.3f' % res1.best_score_)

best learning rate = 0.05
best number of trees = 500
log loss with optimal parameters: -0.501

From the grid search, the model is optimized with learning rate = 0.05 and number of trees = 500. The learning rate plot shows the start of some overfitting past 0.1.

Even though the grid search recommended 500 trees, I'm going to use 300 instead. The additional trees offer fairly marginal improvement to the model while increasing runtime, which doesn't seem like a worthwhile tradeoff, especially since I'm going to eventually need to make predictions for around 500k data points. For reference, here's the top 5 parameters by log loss. Using 300 only reduces the log loss score by 0.004.

Max Tree Depth¶

Max tree depth controls the number of nodes allowed for a tree. sklearn defaults this to 3, so I'll start around there and go as high as around 15.

In [80]:
# set new params to test
grid = {'max_depth': [2, 4, 6, 10, 15]}

# this time, use ideal learning rate and num. trees
model = GradientBoostingClassifier(
    random_state=0,
    learning_rate=0.05,
    n_estimators=300
)

res2 = GridSearchCV(model, grid, scoring='neg_log_loss',
                    return_train_score=True,
                    n_jobs=4).fit(X, y)
results2 = pd.DataFrame(res2.cv_results_)

print('best max_depth = %s' % res2.best_params_['max_depth'])
print('log loss with optimal parameters: %0.3f' % res2.best_score_)

best max_depth = 4
log loss with optimal parameters: -0.497

Similar improvement here as with tuning learning rate and number of trees. Larger max depth makes a real difference for overfitting -- training error heads to 0 at a max depth past 10 while testing error takes a nosedive.

Min samples split¶

min_samples_split governs the number of samples of data needed to split a node into leafs. The sklearn default is 2. I have a pretty large dataset, so this can get large -- in some previous runs, I got an optimal number of samples of as high as 500. However, since at least one of my possible outcomes (triples) are fairly uncommon, I don't want the model to gloss over them by requiring too many samples to split a node, so I'm going to force this to be smaller.

Min samples leaf¶

Finally, we have min_samples_leaf. min_samples_leaf is similar to min_samples_split, requiring a minimum number of samples to create a leaf. The sklearn default is 1.

model = GradientBoostingClassifier(

Adjusting min_samples_leaf looks pretty similar to min_samples_split, which isn't too surprising. Leafs are just nodes that have no additional nodes coming off them -- they make a classification instead of passing to another node.

Tuning Summary¶

Successfully completed hyperparameter tuning! I didn't get a massive improvement in log loss (-0.515 to -0.495), but I'm still happy with the results. Here's the ultimate optimal parameters:

  • learning rate = 0.05
  • number of trees = 300
  • max tree depth = 4
  • min samples to split node = 4
  • min samples to split leaf = 10
